# 第三方口吻外链分发工具

## Problem Frame

编辑（赵老师团队）为单一首发主站做长期 SEO 外链建设。已有 v0.1 流程是 `URL → 爬取 → LLM 改写 → 多平台发布`，但实际工作中编辑通常已经手写好原创草稿，真正需要的是：把草稿"换皮"成**看起来像第三方独立用户写的推荐文**，并分发到多个海外博客平台，制造**自然且具备多样性**的反向链接。

当前 v0.1 不匹配真实工作流的几个点：
- 入口是 URL 抓取，而不是粘贴草稿
- 改写目标是"通顺改写"，不是"伪装成第三方语气推荐特定主站"
- 7 个平台共用同一篇会被 Google 判为重复内容，稀释外链权重
- 缺少首发方品牌词典、锚词词库、人设差异化能力

目标：把工具从"通用 syndicator"重新定位为**单主站、多人设、多变体的第三方推荐外链工厂**，每日中频（3-10 篇）稳定输出。

## Requirements

**输入与首发方资料**
- R1. 编辑入口从"URL 输入"切换为"原文粘贴"。表单字段：原文正文（必填）、本篇标题（可选，AI 可重写）、目标着陆页 URL（可选，覆盖默认）、自定义关键词或备注（可选）。原 URL 抓取流程作为可选辅助保留，不再是主路径。
- R2. 系统维护一个**首发方品牌资料库**（单条记录即可，不做多品牌切换；数据模型预留 `brand_id` 字段，默认 `main`，便于未来扩多品牌时无需 schema 迁移）。字段包含：品牌主名、品牌变体/拼写差异、默认目标页清单（首页/产品页/活动页等多个 URL + 每个 URL 的语境标签，AI 可按文章语境选）、**身份暴露禁用词列表**（如"作为我们""本品牌""我们的产品"等，发布前 regex lint 命中即阻断，见 R23）、**禁用锚词列表**（不允许 LLM 生成的锚文本变体）、必带尾巴（如固定的免责声明或推广话术，可选）。
- R3. 资料库提供后台简易表单维护（不做权限分级，单租户即可），改动立刻生效，无需重启。**前置条件门**：发布前必须满足品牌主名 + 至少 1 个默认目标页 URL；身份暴露禁用词列表 ≥ 5 条（保证 lint 有效）。任一字段缺失时 UI 阻塞提交并提示编辑先填写。

**LLM 改写与人设**
- R4. LLM 改写目标是把原文转成**自然的第三方推荐文**：第一人称从首发方主体切换为外部观察者/用户，避免"官宣"语气，原文事实和数据点不能丢失，不能出现"作为 [品牌名] 的官方"之类暴露身份的表达。
- R5. **每个目标平台独立生成一个变体**（不复用），以最大化避免 Google 重复内容判定。共生成 7 个变体（对应当前 7 个 API 平台）；浏览器自动化平台不在 MVP 自动化范围。
- R6. 人设按平台分组绑定，**MVP 内置 3 个人设组**：
  - 技术博主（Dev.to / Hashnode / GitHub）：偏技术分析、案例拆解、代码或数据
  - 个人随笔（Medium）：第一人称体验、情感、故事化
  - 评论客（Telegraph / Blogger / WordPress）：中立观察、对比同类、推荐结尾
  人设的 prompt 模板独立成文件，便于后续调优，但 MVP 不做"编辑手选人设"。
- R7. LLM 调用走现有 `agent-llm.ts` 的 OpenAI/Gemini 双通道，保留 fallback；7 次调用之间无依赖，可并行（受 API 速率限制约束）。

**锚链注入策略**
- R8. **锚词动态生成**（替代静态词库）：每篇变体的锚词由 LLM 实时生成 1-2 条候选（输入：品牌名 + 本篇文章主题/关键词 + 落地页语境标签 + 禁用锚词列表）。目标是高熵长尾分布，避免"5-15 条小词池被 Penguin 检测出指纹"的风险。生成的锚词需通过 `禁用锚词列表` 过滤；命中即重抽（最多 3 次重抽，仍命中则降级为"裸 URL"）。
- R9. 锚链插入位置由 LLM 根据语境决定（不强制段落），但必须满足约束：每篇变体至少 1 个、至多 2 个出现在正文中（不算结尾签名），不出现在标题，不在连续两段中都插入锚词。
- R10. 落地 URL 从品牌资料库的"目标页清单"中按文章语境匹配（AI 选）；如果文章无明显匹配点，回落到默认主页。
- R10b. **锚词分布监控**：系统统计近 30 篇成功发布的所有锚词使用频次。当任一锚词出现频次 / 总锚词数 > 30% 时，预览页对当前批次显示"锚词集中度提醒"；不阻断发布，但要求编辑明确确认。
- R10c. **单目标 URL 周反链上限软控**（默认 6 篇/周/URL，可配）：编辑提交批次时，如果该批次的"落地 URL × 平台数"会让某条目标 URL 在过去 7 天内累计反链超过上限，UI 提示"已达建议上限"，要求编辑明确确认或改选另一个目标页。不硬阻断（阻断会让工具反复弹窗影响日常工作流）。

**预览审核闸**
- R11. 提交后跳转到预览页，**7 个 tab** 分别展示每个平台的标题、正文 Markdown、锚词、落地 URL、人设标签。
- R12. 每个 tab 支持：单独编辑正文、单独"重新生成"（仅这一个变体重跑 LLM）、"跳过这个平台"（不发布该平台）。
- R13. 全局"一键发布"按钮：把未跳过的变体全部入队。发布后预览数据保存为草稿历史（再次打开可看上次发了什么）。

**队列调度与发布**
- R14. 中频场景（3-10 篇/日，每篇 ×7 平台 = 21-70 个发布作业）需要**作业队列**：在 SQLite 中**新增作业表**（v0.1 仅有 `posts` 和 `task_progress`，无延迟调度表，需新建 `publish_jobs(batch_id, variant_id, platform, scheduled_at, status, attempts, ...)`），按"提交批次"组织。进程重启时需从持久化作业表恢复 `status='scheduled' AND scheduled_at > now()` 的未到期任务。
- R15. 同一批次内，**同平台连发的间隔为随机 N 分钟**（例如 20-90 分钟均匀分布，可配置），避免短时连发触发风控；不同平台之间无强制间隔。
- R16. 单平台失败不影响其他平台：失败标记为可重试，自动重试至多 2 次，仍失败则在 UI/Sheets 标红，不阻塞队列后续作业。**重试在队列层统一编排**（每作业最多 2 次队列级重试，每次重试依然走 R15 的随机间隔）；现有 `publish-service.ts` 内嵌的 retry 循环在重构时收敛为单次 `publishOnce(adapter, payload)`，避免重试次数被乘到 4 次。
- R17. 队列状态可在 UI 看：进行中、待发、已发、失败可重试、已放弃。

**结果追踪**
- R18. 每条发布记录入 SQLite：批次 ID、品牌 ID、平台、人设、锚词、目标 URL、发布 URL、状态、时间、重试次数、错误信息、**T+24h 存活、T+7d 存活**（默认 null，存活检查跑完后回填 200/404/410/timeout 之一）。
- R19. 同步写入 Google Sheets（沿用现有集成），列结构包含品牌、批次、平台、发布 URL、锚词、人设标签、状态、T+7d 存活。Sheets 是编辑日常 canonical 视图。
- R20. 批次详情页降级为**Sheets 深链**（点击批次进入对应 Sheets 行 / 子 sheet），不单独建 UI 页面，避免数据视图三处重复。
- R21. **链接存活检查**：发布成功后 24 小时和 7 天分别对每条 published URL 跑一次 HEAD 请求，结果回写 R18 的存活字段。失效（404 / 410 / 长时间超时）时在 Sheets 该行染色 + 队列状态页"已删帖"分组里展示。失活检查不重发，仅记录。
- R22. **跨批次聚合视图（轻量）**：在 Sheets 中加一个汇总 sheet，按目标 URL / 平台 / 人设维度展示累计反链数和 T+7d 存活率（基于 R18 的字段，纯查询，无新数据采集）。
- R23. **发布前 lint 闸**：每个变体在入队列发布前必须通过两道自动检查——(a) 身份暴露 regex lint（命中 R2 的 "身份暴露禁用词列表" 即阻断该变体，要求编辑修订或跳过）；(b) **5-gram Jaccard 去重检查**（计算该变体与同批次其他变体的 5-gram Jaccard 相似度，任意两两 ≥ 0.4 时阻断该批次，要求至少重生成一个变体）。Lint 是阻断式，不再走 R12 的"事后抽审"。

## Success Criteria

**性能 / 落地**
- 编辑粘贴一篇原文后，**P50 < 90 秒**看到 7 个平台变体预览（LLM 并行下、无 rate-limit 重试时）；**P95 < 180 秒**（容忍一次 rate-limit 退避）。
- 编辑可在预览页**单 tab 重生成**而不影响其他变体。
- 单批次发布的链接落地率 ≥ 90%（剩余 ≤ 10% 在 SQLite 标记为可重试）。
- 在 Google Sheets 上能立刻查到每篇文章在 7 个平台的落地 URL。

**内容质量（自动化、可阻断）**
- **身份暴露**：发布前 regex lint 命中即阻断；R2 禁用词列表覆盖率每月人工审视 1 次（保证不漏新变体）。lint 误放率（人工抽查 30 篇发现身份暴露但 lint 未拦下的比例）≤ 3%。
- **变体差异度**：同一批次任意两个变体的 **5-gram Jaccard 相似度 < 0.4**（本地计算，零外部依赖，发布前阻断）。

**反链图谱健康（新增）**
- **T+7d 链接存活率 ≥ 80%**（按月统计；低于 70% 时自动告警，提示编辑某个平台账号可能被风控）。
- **锚词集中度**：近 30 篇任一锚词出现频次 / 总锚词数 ≤ 30%（违反时仅 UI 提醒，不阻断；月度复盘）。
- **单目标 URL 周反链速率**：默认 ≤ 6 篇/周/URL；越限走 R10c 的明确确认流程，月度复盘所有"被确认越限"的批次。

## Scope Boundaries

- **不做多品牌切换 UI**：单主站资料库 + brand_id 字段预留即可。多品牌切换 UI 延迟到品牌库做大后再说。
- **不做编辑手选人设**：MVP 用平台分组绑定。
- **MVP 不在自动化范围内的 BrowserAutomationAdapter**：MVP 只覆盖 7 个 API 平台，浏览器自动化平台保留代码但不进入"一键发布"队列。
- **不做高频/多账号矩阵**：按中频 3-10 篇/日设计，单账号即可；多账号轮换是后续阶段事项。**接受单账号被封禁的风险**——通过 R21 的存活检查 + 月度存活率告警让风险可观测，由赵老师团队判断是否升级到多账号。
- **不做 SEO 排名监控**：是否被 Google 收录、关键词排名变化不在本工具职责内（建议团队每月人工查 Google Search Console）。
- **做** 反链存活检查（R21）和图谱健康指标（成功标准章），这是 MVP 的 P0 防御。
- **不做中文平台**：现有 7 个英文平台保持不变。
- **不做权限/多用户**：单租户内部工具。
- **不做 Captcha/反风控对抗**：依赖现有平台 API 鉴权 + 随机间隔 + 速率软控，不做主动反检测；如果某平台开始要求 Captcha，从 MVP 中剔除该平台而不是对抗它。
- **不防御 Google 算法重大变更**：本工具基于 2026 年初公开的反作弊信号假设构建；如果 Google 推出新型 AI 反链识别，需要重新评估工具策略，可能性视作业务风险接受。

## Key Decisions

- **每平台独立重写**（R5）：明确接受 LLM 成本 ×7，换取更干净的反链多样性。Planning 阶段会做 A/B 验证（独立重写 vs base+tone-shift），如果后者也能过 Jaccard < 0.4 阈值，可以省成本——但 MVP 默认走独立重写。
- **人设按平台组绑定**（R6）：3 组覆盖现有 7 平台，prompt 文件化方便后续调优；Planning 阶段决定是否 MVP 简化到 1 个通用第三方人设起步。
- **锚词动态生成（替代固定词库） + AI 选位 + 分布监控**（R8/R9/R10b）：放弃 5-15 条静态词库（footprint 过小），转 LLM 长尾生成 + 禁用列表过滤 + 分布监控三层防御。这是基于 adversarial review 后的关键调整。
- **发布前 lint 阻断，预览仅做事实/语气校对**（R23 + R11）：lint（身份暴露 + Jaccard 去重）由系统自动阻断，编辑预览主要检查事实和语气，避免 21-70 tabs/天压垮人工。
- **接受发布期风险、监控存活率反向校准**（R21 + Scope Boundaries）：MVP 不做多账号、不防 Captcha，但加上 T+24h/T+7d 存活检查。如果月度存活率跌破 70% 触发告警，再考虑升级（不在 MVP 范围）。
- **Sheets 是编辑日常 canonical，砍掉重复 UI**（R19/R20）：批次详情页降级为 Sheets 深链；SQLite 仅持久化；UI 只展示队列状态。
- **批次内随机间隔，跨平台不限速**（R15）：把限速放在"同平台同账号"维度，最贴合风控触发点。
- **do-nothing baseline 假设记录**：本工具假设"中频 3-10 篇/日 + 7 平台分发 + 反链速率软控"对 SEO 净收益 > 编辑团队当前的纯手工放置；该假设未做对照实验，由赵老师团队基于业务判断接受。Planning 阶段建议加一个"上线后 3 个月对比 GSC 排名变化"的复盘节点。

## Dependencies / Assumptions

- 现有 9 个 adapter 全部已实现且 7 个 API adapter 鉴权可用（v0.1 的前提）。
- OpenAI 和 Gemini API key 已配置且账户额度足够支持每日 30-70 次调用。
- Google Sheets 集成沿用现有 service account，写入权限正常。
- SQLite 作为本地持久化够用（本地 CLI/单机部署，无并发写需求）。
- Playwright 浏览器自动化继续支持现有"通用浏览器 adapter"，但 MVP 不进入自动化队列。

## Outstanding Questions

### Resolve Before Planning
- 无。所有产品决策已收敛。

### Deferred to Planning

**品牌 / 人设 / 提示词**
- [Affects R2][User decision] 身份暴露禁用词列表 + 禁用锚词列表的初版条目？需要赵老师团队提供。建议至少 10 条暴露禁用词（"作为我们/本品牌/我们的产品/官方推荐/敝公司"等），3-5 条锚词禁用词（已被滥用的话术）。
- [Affects R6][Needs research] 三个人设组的 prompt 模板初版怎么写？是否需要给每组准备 2-3 个 few-shot 示例？是否 MVP 简化到 1 个通用人设起步、证据出来再分组？
- [Affects R5][Decision] 是否做 A/B 实验对比"7 篇独立重写"vs"1 base + 6 tone-shift"，证据决定是否真的需要 ×7 LLM 成本？

**LLM 调用**
- [Affects R5][Technical] 7 个 LLM 调用并行时是否会撞 OpenAI/Gemini 的 rate limit？Planning 时确认每个变体平均 token 数和并发上限，决定是否需要并发节流（例如 3 路并行而非 7 路全并行）。
- [Affects R8][Technical] 锚词动态生成怎么 prompt？是否需要给 LLM 看锚词分布历史（避免重复抽到同一锚）还是只靠生成时的随机性？
- [Affects R12][Technical] "单 tab 重新生成"的锚词与分布约束如何兼容：是重新抽锚还是保留原锚？重抽时是否要回避同批次其他变体已用的锚？
