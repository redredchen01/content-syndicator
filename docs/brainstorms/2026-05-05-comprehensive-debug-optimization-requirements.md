# 全面 Debug 优化体系建设

## 问题框架

当前项目开发效率受四大痛点制约：
1. **Bug 定位缓慢** — 日志系统过于简陋（仅 console.log，无时间戳、模块追踪、持久化），排查问题需大量手工打日志
2. **测试覆盖不完整** — 有 vitest 框架但缺少系统化的测试覆盖规划，新功能易伤害既有模块
3. **模块交互复杂** — 13 个核心模块（adapters、services、routes 等）职责边界不清、依赖关系混乱，新增功能常产生意外耦合
4. **本地开发成本高** — 缺乏开发工具、调试能力、本地集成测试环节

这四个问题的根源是**架构边界不清**。解决策略：先整清模块关系，配套建设日志、测试、开发工具体系。

## 核心优化方向

### 阶段 1：模块治理 (Foundational)

明确模块边界、梳理依赖关系、建立通信约定。

**R1. 模块职责清单**
- 现有 13 个模块各自的单一职责是什么？（不过早分工，而是明确现状）
- 例：adapters 专职转换；services 专职业务逻辑编排；routes 专职 API 接口
- 文档化在 `ARCHITECTURE.md`

**R2. 依赖关系图**
- 梳理模块间的依赖（谁依赖谁、依赖什么）
- 识别循环依赖、过多跨界依赖（红旗项）
- 用文本或图表形式记录在 `docs/module-graph.md`

**R3. 通信边界定义**
- 定义模块间的通信方式：直接调用 vs 事件发射 vs 中介服务
- 特别是 adapters ↔ services ↔ routes 的数据流向规范
- 预防模块间传参无序、返回值格式不一致

**R4. 单向依赖原则**
- 确保依赖方向指向核心（utils → services → routes），避免反向依赖
- 若 utils 依赖 services 是代码坏味道

---

### 阶段 2：日志系统现代化

在明确模块边界后，日志系统就能清晰地跟踪跨模块调用。

**R5. 结构化日志框架**
- 升级现有 logger 为结构化（时间戳、模块名、调用链 ID、日志级别、上下文字段）
- 推荐轻量库：pino（性能好）或 winston（功能完整）；或自实现简版本
- 输出格式：`[TIME] [LEVEL] [MODULE:FUNCTION] [CONTEXT] message`
- 例：`[2026-05-05T10:30:45.123Z] [DEBUG] [services:publishContent] [context_id=abc123] Processing publish for adapter: Blogger`

**R6. 日志文件持久化**
- 日志输出到 `./logs/{date}.log`，按日期分割
- 支持日志级别过滤（本地开发默认 DEBUG，生产默认 INFO）
- 实现简单日志轮转（避免文件无限增长）

**R7. 关键路径埋点**
- 在模块交界处埋点：adapters 入出口、services 关键业务逻辑、routes 全量请求/响应
- 特别关注 voice syndicator、sheets 聚合、scheduler 等近期新增模块的日志覆盖

**R8. 上下文追踪**
- 为每个请求/任务自动生成 context_id（trace_id），贯穿整个调用链
- 使代码快速跳转到某个具体请求涉及的所有日志行

---

### 阶段 3：测试覆盖体系

明确模块职责后，才能定义清晰的单元测试边界和集成测试场景。

**R9. 单元测试覆盖规划**
- 按模块定义单元测试覆盖目标：核心业务逻辑（services）目标 80%，adapters 目标 70%，utils 目标 90%
- 编写单元测试时严格 mock 外部依赖（数据库、外部 API、其他模块）

**R10. 集成测试框架**
- 建立本地集成测试：测试 adapters → services → routes 的完整调用链
- 为常见场景编写集成测试：发布流、数据同步流、错误恢复流
- 每个集成测试对应一个 user story 或痛点场景

**R11. 模块间通信的测试**
- 如果 services 依赖多个 adapters，需测试 adapters 间的协作
- 如果 scheduler 与其他模块通信，需测试不同时序下的行为

**R12. 回归测试清单**
- 近期新增的 voice syndicator、sheets 聚合、scheduler 各有清单化的回归测试
- 每次改动前后都运行一遍，预防伤害

---

### 阶段 4：本地开发工具链

配套提升本地开发体验。

**R13. 启动脚本优化**
- 简化本地启动（`npm start` 一键启，不需要手动启动依赖）
- 或提供清晰的依赖启动指南 + preflight 检查（健康度检查）

**R14. 调试辅助工具**
- 本地日志查看工具：快速搜索、过滤、追踪 context_id
- 或提供 VSCode 调试配置（断点调试）

**R15. 开发者文档**
- 新人上手文档：模块地图、API 调用约定、常见调试场景的排查步骤
- 例：如何跟踪一个发布请求从 routes → services → adapters 的完整路径

- [ ] 核心业务模块（services）单元测试覆盖 ≥ 80%

- [ ] 本地启动流程已优化，新人能 5 分钟内启动项目
- [ ] 下一个 bug 报告能在 30 分钟内通过日志定位到根因（目前通常需要 2 小时+）

## 范围界限

**不在本轮范围内：**
- 模块重构或架构大改（只是梳理现状，为未来改进铺路）
- 完整的容器化或微服务拆分
- 性能优化（这是后续工作）
- UI 测试或 E2E 测试（focus 在服务层）

---

## 关键决策

| 决策项 | 选择 | 理由 |
|------|------|------|
| 日志库选型 | 自实现简版 or 轻量库(pino) | 项目规模不大，自实现可控；或 pino 分性能好 |
| 测试框架 | 继续用 vitest | 现有配置完整，无需换库 |
| 依赖管理模式 | 明确模块间调用约定，暂不引入 DI 容器 | 项目规模和复杂度暂不需要 DI，约定优于框架 |
| 部署策略 | 原有策略不变，只加本地开发工具 | 避免触发部署链路变化 |

## 依赖和假设

- **假设：** 现有 13 个模块无法一次完全重写，需要渐进式改进
- **假设：** 大部分模块间依赖是合理的，只是缺乏清晰文档
- **依赖：** 团队同意新编写的测试规范，及后续编码遵循
- **依赖：** 开发环境支持 Node.js 18+ 和 TypeScript（已满足）
